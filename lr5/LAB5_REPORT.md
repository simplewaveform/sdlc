# Лабораторная работа №5  
## «Технический долг. Устранение долга»

---

## 1. Кратко по теории (выжимка из статей)

- **Как «пощупать» техдолг быстро:** команда субъективно оценивает качество кода по шкале 1–5 и обсуждает расхождения оценок; визуализация проблемы — первый шаг к решению (идея из *The Solution to Technical Debt*).
- **Хороший и плохой долг:**
  - «Свежий беспорядок» в ходе реализации новой фичи может быть **хорошим долгом** (быстрое исследование решения/прототип).
  - Долг становится **плохим**, если его не «погасить» вскоре (идеально — «пила»: накопили во время фичи → почистили до следующей) (*Good and Bad Technical Debt*).
- **Definition of Done как ограничитель долга:** общие критерии приёмки («код чистый», «в продакшене», «принято пользователем/по сценариям») позволяют удерживать долг в допустимых пределах (идея из PDF).
- **ATDD как инструмент погашения долга:** начать с описания пользовательского потока, превратить его в исполнимый acceptance-тест, добиться падения, затем прохождения, и в конце — почистить реализацию (*ATDD From the Trenches*).

## 2. Инспекция проекта на признаки технического долга

Ниже — проверка по пунктам из задания (с примерами из текущего репозитория).

### 2.1 Непонятный / нечитабельный код

- **Наблюдение:** код в целом читаемый (проект небольшой), но есть «скрытые» допущения и смешение ответственностей.
- **Примеры:**
  - `backend/src/routes/auth.js`: `requireAuth` объявлен внутри роутера; в этом же файле и бизнес-правила, и работа с БД, и обработка ошибок.
  - `frontend/src/components/Layout.jsx`: обмен состоянием авторизации через `window.dispatchEvent(new Event('authChange'))` — рабочее решение, но неочевидное и хрупкое при росте приложения.
- **Риск:** при расширении функционала (роли/права, бронирования) будет сложнее тестировать и поддерживать.

### 2.2 Дублирующийся код

- **Наблюдение:** есть повторяющиеся шаблоны.
- **Примеры:**
  - Повтор `try/catch + console.error + res.status(500)` в нескольких ручках (`cars.js`, `auth.js`).
  - На фронте повтор обработки `res.ok` и вытаскивания `data.error` в `frontend/src/api/auth.js` (аналогичное есть и в `api/cars.js`).
- **Риск:** правки поведения ошибок/формата ответа потребуют менять несколько мест.

### 2.3 Отсутствие автоматизации (тестов, сборки, развёртывания)

- **Наблюдение:** тестов нет, линтинга/форматтера нет, CI нет.
- **Факты в репозитории:**
  - `backend/package.json`: отсутствуют скрипты `test`, зависимости для тестов/линтера.
  - `frontend/package.json`: отсутствуют `test`, `lint`, `format`.
  - Есть `start.sh`, но он не заменяет CI и тесты.
- **Риск:** регрессии ловятся вручную; качество падает при росте команды и объёма кода.

### 2.4 Запутанная архитектура и ненужные сложные зависимости

- **Наблюдение:** зависимостей мало (это плюс), но архитектурно код «плоский» — бизнес-логика живёт рядом с HTTP и SQL.
- **Факты:**
  - Прямые SQL-запросы из роутов (`backend/src/routes/*.js`), нет выделенного слоя сервисов/репозиториев.
- **Риск:** сложно добавлять сложные сценарии (бронь с проверками доступности/правами).

### 2.5 Медленные / неэффективные средства

- **Наблюдение:** в текущем масштабе явных проблем нет, но «узкие места» потенциально появятся.
- **Факты/риск:** отсутствие профилирования и метрик; с ростом данных запрос `SELECT * FROM cars ...` без явных индексов/пагинации может стать проблемой.

### 2.6 Незакоммиченный код / долгоживущие ветки

- **Наблюдение:** в текущей модели разработки работаем в основном в `main`. Риск «долгоживущих веток» возникает, если появятся параллельные фичи без CI/тестов.
- **Потенциальный долг:** без практик trunk-based или коротких веток + PR/проверок интеграция может «застревать».

### 2.7 Отсутствие / несоответствие технической документации

- **Наблюдение:** базовый `README.md` есть (запуск, первый спринт), но документации для API/архитектуры/DoD/плана качества нет.
- **Риск:** знания остаются «в головах», сложнее онбординг и контроль качества.

### 2.8 Отсутствие тестовой среды

- **Наблюдение:** отдельной тестовой среды нет.
- **Факты:** SQLite-файл создаётся локально, но нет разделения `dev/test`, нет фикстур для тестов.
- **Риск:** невозможно безопасно гонять автотесты и проверять миграции/seed.

### 2.9 Длинные циклы интеграции / отсутствие CI

- **Наблюдение:** CI отсутствует → интеграция «по факту» ручная.
- **Риск:** чем больше задач параллельно, тем выше вероятность конфликтов и регрессий.

## 3. План мероприятий по устранению техдолга

### 3.1 Ввести «границы долга» (Definition of Done)

Предлагаемый DoD для задач, которые попадают в основную ветку:

- Функциональность реализована и проходит **пользовательские сценарии** (ручные сейчас, автотесты — далее).
- Есть обработка ошибок и понятные сообщения на UI.
- Нет «висящего» временного кода (закомментированных блоков/заглушек без задач в плане).
- Документация обновлена (минимум: README/описание эндпоинтов).

### 3.2 Автоматизация качества

- **Линтинг/форматирование:** добавить ESLint + Prettier для `frontend` и `backend`, запуск в pre-commit (или хотя бы скриптами).
- **Тесты (минимальный слой):**
  - backend: интеграционные тесты API (например, Supertest) для `/api/auth/*` и `/api/cars/*`;
  - frontend: e2e-тесты (например, Playwright) на ключевые сценарии.
- **CI:** GitHub Actions (или аналог) — `install → lint → test → build`.

### 3.3 ATDD-план (что именно тестировать)

Acceptance-сценарии (минимум, чтобы «застолбить» поведение и перестать бояться рефакторинга):

- **Каталог:**
  - открыть `/` → виден список машин;
  - фильтр по `model` и `maxPrice` меняет выдачу;
  - переход на `/cars/:id` работает; 404 показывает корректное состояние.
- **Учётные записи:**
  - регистрация → вход → переход в профиль;
  - `GET /me` без токена → 401;
  - обновление профиля меняет отображаемое имя в шапке;
  - удаление аккаунта делает сессию недействительной.

ATDD-цикл по статье: описываем поток → делаем тест (сначала падает) → доводим до прохождения → чистим код.

### 3.4 Снижение архитектурного долга (точечно, без «переписывания»)

- Выделить **middleware** (`requireAuth`) в отдельный модуль.
- Вынести работу с БД (SQL) в простой слой репозиториев (минимум: `usersRepo`, `carsRepo`).
- Унифицировать обработку ошибок (один error-handler + единый формат `{ error, code? }`).

### 3.5 Документация и тестовая среда

- Добавить `.env.example` (например, `JWT_SECRET`, `DATABASE_PATH`).
- Описать API в `README.md` (эндпоинты и примеры запросов).
- Развести БД/seed для `dev` и `test` (отдельные файлы SQLite, фикстуры).

## 4. Оценка плана (грубая)

Оценка дана в человеко-часах для небольшого проекта (без «идеального» рефакторинга).

| Блок | Что делаем | Оценка |
|---|---|---:|
| DoD | сформулировать и зафиксировать в README | 1–2 ч |
| Линт/формат | ESLint+Prettier, скрипты | 2–4 ч |
| Backend tests | Supertest: auth + cars | 4–8 ч |
| Frontend e2e | Playwright: 4–6 сценариев | 6–12 ч |
| CI | GitHub Actions: lint/test/build | 1–3 ч |
| Арх. долг (точечно) | вынести middleware + repo-слой + единый error-format | 6–12 ч |
| Документация | `.env.example` + описание API | 2–4 ч |

Итого: **~22–45 ч** (в зависимости от глубины тестов и рефакторинга).

## 5. Сравнение: объём долга vs недоимплементированные фичи

### 5.1 Долг (крупными блоками)

1) Нет автоматизированных тестов (backend + e2e)  
2) Нет CI (проверок на интеграции)  
3) Нет линта/форматтера (единых стандартов кода)  
4) Плоская архитектура (роуты = бизнес-логика = SQL)  
5) Недостаточная документация (API/настройки/DoD)  

### 5.2 Недореализованные фичи (из плана спринта по учётным записям)

- Назначение ролей (SCRUM-32)  
- Проверка прав на действия (SCRUM-34)  
- Восстановление/сброс пароля (SCRUM-28)  

### 5.3 Вывод сравнения

По числу крупных пунктов **долг сопоставим с количеством оставшихся фич**. При этом часть долга (тесты+CI+линт) напрямую снижает риск регрессий при реализации ролей/прав/сброса пароля, поэтому его выгодно погасить **до** усложнения функционала.

## 6. Корректировка плана разработки проекта

Предлагаем баланс: **не «останавливать» продукт ради идеального рефакторинга**, но погасить долг, который:
1) уменьшает риск поломок при добавлении фич,  
2) ускоряет работу команды (быстрее проверка, меньше ручной рутины).

### 6.1 Предлагаемый план (следующие 1–2 спринта)

**Спринт A (качество + безопасность основы):**
- Ввести DoD и минимальную техдокументацию (README: API/настройки).
- ESLint+Prettier и единые скрипты.
- CI: lint + тесты + build.
- Минимальные тесты на auth/cars (backend) и 2–3 e2e сценария (критический путь).

**Спринт B (фичи + закрытие долга точечно):**
- Реализация ролей и проверок прав (SCRUM-32/34) **под защитой тестов**.
- Добавление e2e на права/роли.
- Точечный рефакторинг: вынести middleware/репозитории (только там, где мешает развитию).

**Далее (после стабилизации):**
- Восстановление/сброс пароля (SCRUM-28).
- Расширение тестового покрытия и архитектурный рефакторинг по необходимости (не раньше, чем появится реальный «больной» участок).

## Итог

Инспекция показала, что основной технический долг сейчас — **отсутствие автоматизации качества (тестов, линта, CI) и плоская архитектура**, которая начнёт мешать при росте фич (роли/права, дальнейшая аренда/бронирования). Оптимальная стратегия — ввести DoD и минимальный ATDD-набор сценариев, поднять CI и только затем активно наращивать функциональность.
